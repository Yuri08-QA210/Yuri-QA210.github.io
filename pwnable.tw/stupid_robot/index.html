2. Static Analysis

The source code reveals a C++ game with a class hierarchy. The critical vulnerability lies in the Web::compatible() and Trojan::compatible() methods, which are used to decide whether to create a new Language object or reinterpret_cast the existing one.

Source Code: The Vulnerable Functions

c++
// The Web project's compatible() function:
bool compatible(LANG lang1, LANG lang2)
{
    string s1 = LANG_NAME[lang1];   // NEW language
    string s2 = LANG_NAME[lang2];   // OLD language

    /* I hate my stupid Boss A */
    if(strstr(s2.c_str(), s1.c_str())) return true;   // ← BUG!
    else return false;
}

// LANG_NAME values:
// "ASM", "C", "Ruby", "Java", "Python", "Javascript"

// When we change Java → Javascript:
// s1 = "Java", s2 = "Java"
// strstr("Java", "Java") → TRUE! (substring match)
// So the code takes the "just_edit" path:

case Java:
{
    L_Java *temp = just_edit == true
        ? reinterpret_cast<L_Java*>(*l)     // ← TYPE CONFUSION!
        : new L_Java(r);
    temp->set_version();
    *l = temp;
    break;
}

// But we're changing FROM Java TO Javascript:
case Javascript:
{
    L_Javascript *temp = just_edit == true
        ? reinterpret_cast<L_Javascript*>(*l)  // ← ACTUALLY AN L_Java!
    temp->set_version();     // Calls L_Javascript::set_version()
    *l = temp;               // on an L_Java object!
    break;
}

Type Confusion: L_Java vs L_Javascript

The core issue is that L_Java and L_Javascript have different member layouts:

c++
class L_Java : public Language {
    // Inherited from Language:
    //   LANG lang;       (4 bytes)
    //   string name;     (32 bytes on x64)
    //   ul ratio;        (8 bytes)
    // + vtable pointer   (8 bytes)
private:
    double version;   // 8 bytes — stores version like 8.0
};

class L_Javascript : public Language {
    // Inherited from Language:
    //   LANG lang;       (4 bytes)
    //   string name;     (32 bytes on x64)
    //   ul ratio;        (8 bytes)
    // + vtable pointer   (8 bytes)
private:
    char *version;    // 8 bytes — POINTER to heap string!
};
double vs char* — The Same 8 Bytes, Two Meanings

When we create a Web project with L_Java, the version field is a double. If we then "change" to Javascript, the compatible() function returns true (because "Java" is a substring of "Javascript"), and the code does reinterpret_cast<L_Javascript*> on the L_Java object. Now the double version bits are interpreted as a char* version pointer! If we control the double value (via scanf("%lf")), we control where L_Javascript::set_version() writes with read(0, version, 20).

Vulnerability Analysis

The full exploitation chain involves two paths through the type confusion:

Path 1: Web Project (Java → Javascript) — Arbitrary Write

  1. Create a Web project with Java language
  2. Set Java version via scanf("%lf") — this stores a double value
  3. Change language to Javascript — compatible() returns true ("Java" ⊂ "Javascript")
  4. The existing L_Java object is reinterpret_cast to L_Javascript*
  5. Calling L_Javascript::set_version() does read(0, version, 20) where version is actually the double bits reinterpreted as a pointer
  6. We have an arbitrary write primitive!

Path 2: Trojan Project (C → Python) — Info Leak

  1. Create a Trojan project with C language
  2. C's version is a char* pointing to a heap string
  3. Change language to Python — compatible() returns true ("C" and "Python" are "similar")
  4. The L_C object is reinterpret_cast to L_Python*
  5. L_Python::info() prints the double version, which is actually the char* bits
  6. We can leak heap/libc addresses!
Boss A's Comment Is the Hint

The code comment /* I hate my stupid Boss A */ and Boss A's dialogue "You know Java? Then you must be familiar with Javascript!" are the in-game hints pointing directly at the vulnerability. The boss thinks Java and Javascript are the same language because of the name similarity — and the strstr check implements exactly this flawed logic.

4. Exploit Strategy

Attack Overview

Stage 1: Skill Grinding
Beat slimes to accumulate 500+ exp in the required languages. For the Web path, we need Ruby ≥ 500 and Java ≥ 500 to unlock the Web project. We also need Javascript ≥ 500 (automatically set after creating Web project) for the type confusion to trigger.
Stage 2: Leak Libc Address
Use the Trojan project path (C → Python type confusion) or the show_info() method to leak heap or libc addresses. When L_Java::info() is called on a type-confused object, it may print pointer values that reveal addresses.
Stage 3: Arbitrary Write
Set the Java version double to an IEEE 754 value whose bit pattern equals a target address. Then change to Javascript to trigger read(0, version, 20) which writes 20 bytes to that address. This gives us a controlled arbitrary write primitive.
Stage 4: Control Flow Hijack
Overwrite a vtable pointer or a function pointer in the Language object. When info() or set_version() is called through the vtable, execution jumps to our controlled address. We build a ROP chain for system("/bin/sh") or similar.

Why This Challenge Matters

Stupid Robot teaches a critical lesson about C++ type safety. The reinterpret_cast is one of the most dangerous casts in C++ — it reinterprets the raw bytes of an object as a completely different type. In this case, a double field is reinterpreted as a char* pointer, giving us an arbitrary write primitive. This pattern appears in real-world codebases where inheritance hierarchies use down-casting without proper type checking (RTTI). The compatible() function's flawed substring check mirrors real bugs where developers make incorrect assumptions about type relationships.
